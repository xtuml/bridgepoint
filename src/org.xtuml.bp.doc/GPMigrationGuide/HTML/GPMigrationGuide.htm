1.    Introduction

The BridgePoint meta-model previously used the strict system organization specified by the Shlaer-Mellor method the tool was originally designed to support. This strict organization works very well, but is at variance with the much less prescriptive approach provided by UML 2 which BridgePoint now supports.

 

As of version 3.3, BridgePoint xtUML supports “generic” packages.  These generic packages may contain any combination of many types of model elements.  The former “specialized” packages (e.g. Subsystem, Functions, Sequence Diagram, etc…) that were restricted to contain only applicable model elements are being deprecated. 

 

This document is intended to help application modelers as well as model compiler writers make the transition from specialized packages to generic packages.  A future release of BridgePoint will disable the creation of specialized packages in the user interface.  A release to follow that will remove specialized packages from the BridgePoint meta-model.

 

2.    xtUML Modeling

The UML editor has changed and been enhanced in a number of ways to support the generic package feature.

Upgrading existing models

In order to ease the transition to generic packages, BridgePoint includes a tool to upgrade existing models.

 

The BridgePoint Model Import action can now be used to convert “.xtuml” format model files to generic packages (Figure 4). During the import process, the model is scanned for specialized packages.  Corresponding generic packages are created and the contained elements are migrated to the appropriate generic packages.

 

Figure 4

 

To use this method to upgrade an existing model, follow these steps:

a)     In the Model Explorer view, right click on the project and select Export…

b)    Navigate the tree to BridgePoint > BridgePoint Model, select Next >

c)     There should be a checkmark next to the selected project, now enter a Destination File Name, select Finish

d)    In the Model Explorer view, right click and select New > xtUML Project

e)     Enter a Project name, select Finish

f)     Right click on the new project, select Import…

g)    Navigate the tree to BridgePoint > BridgePoint Model, select Next >

h)     There should be a checkmark next to the selected project, now use Browse… to locate the previously exported file

i)      Check Convert to generic packages, select Finish

j)      The new project now contains the converted model

k)     Copy marking and hand-craft source files from the old project’s gen/ folder to the new project’s gen/ folder

 

3.    Model Verifier

The process of debugging xtUML applications with Model Verifier is unchanged by the move to generic packages.   The Verifier can handle models that are entirely composed of generic packages or that contain a mixture of generic and specialized packages.  In addition, models that are entirely composed of specialized packages are still supported.  Packages themselves are not executable, but components inside generic packages are executable.

 

Of course, the Session Explorer view will look slightly different when using generic packages.  The icon next to the package name will be the generic package icon instead of one of the various specialized package icons.

4.    Model Compilers

The BridgePoint model compilers do not yet support generic package model.  This support will come soon.  Sites that use in-house model compilers must enhance their tools to support the new meta-model classes and associations for generic packages.  There are two packages in the xtUML meta-model that define generic packages: Element Packaging and Packageable Element.

 

Element Packaging (Figure 7) contains the Package class (EP_PKG) as well as classes to model package nesting and legacy support for specialized packages.  Please note that relationships R1403 and R1404 are no longer used, but are still available for backwards compatibility.  Package nesting is handled via the R8000/R8001 and R8003 relationships.

 

The primary addition to the meta-model for generic packages is the Packageable Element (PE_PE) hierarchy (Figure 8). Packageable Element is a supertype class that may be related to zero or one Package across R8000.  The PE_PE relationship R8001 connects the supertype to all the possible elements that may live in generic package.  Of course, PE_PE and EP_PKG are related

5.    Element Visibility

Enhanced ability to control element visibility is now available.  BridgePoint already included some visibility support, which is now clarified and simplified.  Visibility rules apply to Data Types, to Model Classes, to Packages, and to Components.  On the UI canvases, elements whose access is controllable shall be prefixed with + (public), - (private) and # (protected).

 

The visibility rules currently state that package visibility affects the visibility of elements declared inside it. This is incorrect and leads to various unexpected consequences, such as elements being visible when it would be expected that they are not. The visibility rules are simplified by this work to be easier to understand by being more regular throughout the hierarchy.

 

Generic packages now support the UML standard 'protected' visibility type.  The UML standard 'private' semantic restricts access to private elements from child packages whereas the existing BridgePoint 'private' semantic is that child package elements can access elements declared as private when ascending the package hierarchy.

 

Name-spacing is supported and managed entirely by means of visibility. That is, two elements with the same name are permitted provided that they are not simultaneously public in any action language scope. More than one type found during parsing shall be indicated by a parse error.
