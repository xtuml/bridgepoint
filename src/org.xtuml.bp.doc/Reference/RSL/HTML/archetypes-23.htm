Fragment Translation Functions

When .al_xlate is called, the rule file that is generated makes calls to fragment translation functions according to the content of the action under translation. The definitions for the fragment translation functions are provided by the BridgePoint UML Suite model compilers. Each of the functions enumerated below is defined in the rule file fragment.arc found in the model compiler installation.

A fragment translation function exists for each OAL statement and statement component. Functions also exist for statement blocks, and the action itself. The role of some fragment translation functions is to build a fragment from scratch, i.e., from string parameters and xtUML database lookups. The role of other fragment translation functions is to assemble small fragments into larger fragments to eventually create all of the code for an action.

Fragment translation functions have been grouped into the following categories:

  1. Action
  2. Statement Block
  3. Statement
  4. Rvalue
  5. Instance Chain
  6. Parameter List
  7. Variable
  8. Where Clause

Each type of function is discussed in the following sections.

Action Translation Functions

An action is really a block, however, special handling may be required, i.e., declaration statements may need to be output at the beginning of the action only. The action translation functions provide the mechanism for this potential special handling.

actn_begin (p_actn)

This is the first function invoked when the .al_xlate primitive is called on a state action. It begins translation of the action of the state ( SM_ACT ) instance passed in as p_actn .

actn_bridge_begin (p_bridge)

This is the first function invoked when the .al_xlate primitive is called on a bridge. It begins translation of the action of the bridge ( S_BRG ) instance passed in as p_bridge .

actn_synch_service_begin (p_sync_service)

This is the first function invoked when the .al_xlate primitive is called on a function. It begins translation of the action of the function ( S_SYNC ) instance passed in as p_sync_service .

actn_trans_begin (p_trans)

This is the first function invoked when the .al_xlate primitive is called on a class-based operation. It begins translation of the action of the operation ( O_TFR ) instance passed in as p_trans .

actn_ib_trans_begin (p_trans)

This is the first function invoked when the .al_xlate primitive is called on an instance-based operation. It begins translation of the action of the operation ( O_TFR ) instance passed in as p_trans .

actn_attribute_begin (p_attr)

This is the first function invoked when the .al_xlate primitive is called on a derived attribute. It begins translation of the action for the attribute ( O_DBATTR ) instance passed in as p_attr .

actn_append_blck (p_actn, p_blck)

This function is invoked when a block p_blck has been created and needs to be added to the existing action p_actn .

actn_end (p_actn)

This function is invoked when an action instance has been completely translated. The return coordinate is the rule that invoked .al_xlate primitive on this action instance. Therefore, actn_end simply needs to render this action body (i.e., p_actn.body ) to the invoking rule.

blck_begin

This function is invoked when a new OAL block is encountered within the action currently being translated.

blck_append_stmt (p_blck, p_stmt)

This function is invoked when translation of an OAL statement p_stmt has been completed and it needs to be appended to the current block p_blck . The generated code for the statement is contained in p_stmt.body , produced by an stmt_* function.

blck_var_out_of_scope (p_blck, p_var)

This function is invoked when an OAL variable p_var has gone out of scope during the translation of the associated OAL block p_blck . The results of this fragment are passed into the blck_end function.

blck_end

This function is invoked when an OAL block translation has been completed.

stmt_select_from_instances_of (p_cardinality, p_select_var, p_implicit_declare, p_obj_kl)

This function is invoked when a (non-where) selection from an instance collection is encountered in the OAL. The generalized OAL signature is:

select <p_cardinality> <p_select_var> from instances of <p_obj_kl>;

For example: select many dog_set from instances of DOG;

The parameter p_implicit_declare is TRUE if this is the first usage of the object instance reference variable represented by the p_select_var fragment.

stmt_select_from_instances_of_where (p_cardinality, p_select_var, p_implicit_declare, p_obj_kl, p_where)

This function is invoked when a 'where' clause qualified selection from an instance collection is encountered in the OAL. The generalized OAL signature is:

select <p_cardinality> <p_select_var> from instances of <p_obj_kl> where <p_where>;

For example: select many dog_set from instances of DOG where (selected.Weight < 100);

The parameter p_implicit_declare is TRUE if this is the first usage of the object instance reference variable represented by the p_select_var fragment.

stmt_select_related_by (p_cardinality, p_select_var, p_implicit_declare, p_chain)

This function is invoked when a (non-where) selection via association navigation is encountered in the OAL. Note that chain_begin , chain_add_link (s), and chain_end functions have already been invoked to formulate the p_chain fragment prior to this function being called.

The generalized OAL signature is:

select <p_cardinality> <p_select_var> related by <p_chain>;

For example: select many dog_set related by cat->DOG[R1];

The parameter p_implicit_declare is TRUE if this is the first usage of the object instance reference variable represented by the p_select_var fragment.

stmt_select_related_by_where (p_cardinality, p_select_var, p_implicit_declare, p_chain, p_where)

This function is invoked when a where-clause qualified selection via association navigation is encountered in the OAL. Note that chain_begin , chain_add_link (s), and chain_end functions have already been invoked to formulate the p_chain fragment prior to this function being called.

The generalized OAL signature is:

select <p_cardinality> <p_select_var> related by <p_chain> where <p_where>;

For example: select many dog_set related by cat->DOG[R1] where (selected.Weight < 100 );

The parameter p_implicit_declare is TRUE if this is the first usage of the object instance reference variable represented by the p_select_var fragment.

stmt_create_obj_inst_no_var (p_obj_kl)

This function is invoked when a synchronous create of an instance of a class is encountered in the OAL. This function is invoked when the handle to the created instance is not used or assigned to any OAL variable. The generalized OAL signature is:

create object instance of <p_obj_kl>;

For example: create object instance of DOG;

stmt_create_obj_inst (p_inst_ref_var, p_implicit_declare, p_obj_kl)

This function is invoked when a synchronous create of an instance of a class is encountered in the OAL where the handle to the created instance is assigned to the p_inst_ref_var fragment. The generalized OAL signature is:

create object instance <p_inst_ref_var> of <p_obj_kl>;

For example: create object instance this_dog of DOG;

The parameter p_implicit_declare is TRUE if this is the first usage of the object instance reference variable represented by the p_inst_ref_var fragment.

stmt_delete_obj_inst (p_inst_ref_var)

This function is invoked when a synchronous delete of an instance of a class is encountered in the OAL, given the handle to the existing instance, p_inst_ref_var . The generalized OAL signature is:

delete object instance <p_inst_ref_var>;

For example: delete object instance this_dog;

stmt_relate (p_left_inst_var, p_right_inst_var, p_rel_number, p_rel_phrase)

This function is invoked for OAL that relates two instances in a non-associative-link association p_left_inst_var and p_right_inst_var across association number p_rel_number specifying a direction of the association using p_rel_phrase in case of reflexive associations. The generalized OAL signature is:

relate <p_left_inst_var> to <p_right_inst_var> across <p_rel_number>[.'<p_rel_phrase>'];

For example: relate this_dog to that_cat across R1;

stmt_relate_using (p_left_inst_var, p_right_inst_var, p_rel_number, p_assoc_inst_var, p_rel_phrase)

This function is invoked for OAL that relates the three instances in an associative-link association p_left_inst_var , p_right_inst_var , and p_assoc_inst_var across association number p_rel_number specifying a direction of the association using p_rel_phrase in case of reflexive associations. The generalized OAL signature is:

relate <p_left_inst_var> to <p_right_inst_var> across <p_rel_number>[.'<p_rel_phrase>'] using <p_assoc_inst_var> ;

For example: relate this_dog to that_cat across R2 using cat_dog_chase;

stmt_unrelate (p_left_inst_var, p_right_inst_var, p_rel_number, p_rel_phrase)

This function is invoked for OAL that unrelates two instances in a non-associative-link association p_left_inst_var and p_right_inst_var across association number p_rel_number specifying a direction of the association using p_rel_phrase in case of reflexive associations. The generalized OAL signature is:

unrelate <p_left_inst_var> from <p_right_inst_var> across <p_rel_number>[.'<p_rel_phrase>'];

For example: unrelate this_dog from that_cat across R1;

stmt_unrelate_using (p_left_inst_var, p_right_inst_var, p_rel_number, p_assoc_inst_var, p_rel_phrase)

This function is invoked for OAL that unrelates three instances in an associative-link association p_left_inst_var , p_right_inst_var, and p_assoc_inst_var across association number p_rel_number specifying a direction of the association using p_rel_phrase in case of reflexive associations. The generalized OAL signature is:

unrelate <p_left_inst_var> from <p_right_inst_var> across <p_rel_number>[.'<p_rel_phrase>'] using <p_assoc_inst_var>;

For example: unrelate this_dog from that_cat across R2 using dog_cat_chase;

stmt_generate_obj_inst (p_evt_label, p_param, p_inst_ref_var)

This function is invoked for OAL that generates an event to a given class instance. By the time this function is called, the fragment function param_begin_evt_obj_inst has already been invoked as well as the parameter functions param_add and param_end which associate the corresponding event data parameters. The generalized OAL signature is:

generate <p_evt_label>(<p_param>) to <p_inst_ref_var>;

For example: generate DOG1:'Chase Cat'(time:400) to this_dog;

stmt_generate_assigner (p_evt_label, p_param, p_obj_kl)

This function is invoked for OAL that generates an event to a given class assigner (class-based state machine). The fragment function param_begin_evt_assigner has already been invoked as well as the parameter functions param_add and param_end which associate the corresponding event data parameters.

The generalized OAL signature is:

generate <p_evt_label>(<p_param>) to <p_obj_kl> assigner;

For example: generate Chase1:'Assign Dog to Cat'() to Chase Assigner;

stmt_generate_creation (p_evt_label, p_param, p_obj_kl)

This function is invoked for OAL that generates a creation event for a given class. The fragment function param_begin_evt_creation has already been invoked as well as the parameter functions param_add and param_end which associate the corresponding event data parameters.

The generalized OAL signature is:

generate <p_evt_label>(<p_param>) to <p_obj_kl> creator;

For example: generate Cat1:'Begin Life'() to CAT creator;

stmt_generate_ext_entity (p_evt_label, p_param, p_ee_kl)

This function is invoked for OAL that generates an event to an external entity. The fragment function param_begin_evt_ext_entity has already been invoked as well as the parameter functions param_add and param_end which associate the corresponding event data parameters. The generalized OAL signature is:

generate <p_evt_label>(<p_param>) to <p_ee_kl>;

For example: generate EE1:'Event to External Entity'() to EE;

stmt_generate_evt_inst (p_evt_inst_var)

This function is invoked for OAL that generates an event using an already created event instance variable. The generalized OAL signature is:

generate <p_evt_inst_var>;

For example: generate event_one;

stmt_generate_evt_inst_rval (p_evt_inst_var_rval)

This function is invoked for OAL that generates an event using an already created event instance rval. The generalized OAL signature is:

generate <p_evt_inst_var_rval>;

For example: generate dog.event_one;

stmt_create_evt_obj_inst (p_evt_inst_var, p_implicit_declare, p_evt_label, p_param, p_obj_inst_ref_var)

This function is invoked for OAL that pre-creates an instance of an event p_evt_inst_var directed to a given class instance p_obj_inst_ref_var . The generalization OAL signature is:

create event instance <p_evt_inst_var> of <p_evt_label>(<p_param>) to <p_obj_inst_ref_var>;

For example: create event instance e1 of DOG1:'Chase Cat'() to this_dog;

stmt_create_evt_assigner (p_evt_inst_var, p_implicit_declare, p_evt_label, p_param, p_obj_kl)

This function is invoked for OAL that pre-creates an instance of an event p_evt_inst_var directed to the class state machine specified by the key letters p_obj_kl . The generalization OAL signature is:

create event instance <p_evt_inst_var> of <p_evt_label>(<p_param>) to <p_obj_kl> assigner;

For example: create event instance e1 of Chase1:'Assign Dog to Cat'() to Chase assigner;

stmt_create_evt_creation (p_evt_inst_var, p_implicit_declare, p_evt_label, p_param, p_obj_kl)

This function is invoked for OAL pre-creates an instance of an event p_evt_inst_var directed to the class creator specified by the key letters p_obj_kl . The generalization OAL signature is:

create event instance <p_evt_inst_var> of <p_evt_label>(<p_param>) to <p_obj_kl> creator;

For example: create event instance e1 of Cat1:'Begin Life'() to CAT creator;

stmt_create_evt_ext_entity (p_evt_inst_var, p_implicit_declare, p_evt_label, p_param, p_ee_kl)

This function is invoked for OAL that pre-creates an instance of an event p_evt_inst_var directed to the external entity specified by p_ee_kl .

create event instance <p_evt_inst_var> of <p_evt_label>(<p_param>) to <p_ee_kl>;

For example: create event instance e1 of EE1:'Event to External Entity'() to EE;

stmt_assign_obj_attr (p_inst_ref_var, p_attr_name, p_expression_rval)

This function is invoked for OAL that assigns the r-value in p_expression_rval to an attribute of a given class instance specified by <p_inst_ref_var>.<p_attr_name> . The generalization of the OAL is:

[assign] <p_inst_ref_var>.<p_attr_name> = <p_expression_rval>

For example: dog.Name = "Fido";

stmt_assign_transient_var (p_transient_var, p_implicit_declare, p_expression_rval)

This function is invoked for OAL where a transient variable p_transient_var is assigned the r-value p_expression_rval . If this is the first time used, i.e., p_implicit_declare == TRUE, then the transient variable should be declared. The generalization of the OAL is:

[assign] <p_transient_var> = <p_expression_rval>

For example: t1= x+7;

stmt_if (p_condition_rval, p_if_blck)

This function generates the code for the OAL if statement. The conditional expression for the if statement is in the fragment p_conditional_rval (produced by an rval_* function). All the statements contained in the block associated with the if have been processed when this function is invoked. The generated code is contained in p_if_blck.body produced by blck_end function. The generalization of the OAL is:

if ( <p_condition_rval> )

For example: if ( dog.Weight > 100 )

stmt_elif (p_condition_rval, p_elif_blck)

This function generates the code for the OAL elif statement. The conditional expression for the elif statement is in the fragment p_conditional_rval (produced by an rval_* function). All the statements contained in the block associated with the elif have been processed when this function is called. The generated code is contained in p_elif_blck.body (produced by blck_end function). The generalization of the OAL is:

elif ( <p_condition_rval> )

For example: elif ( dog.Weight < 50 )

stmt_else (p_else_blck)

This function generates the code for the OAL else statement and places it in the output fragment attribute body . All the statements contained in the block associated with the else have been processed when this function is called. The generated code is contained in p_else_blck.body (produced by blck_end function). The generalization of the OAL is:

else

stmt_for (p_inst_ref_var, p_implicit_declare, p_inst_ref_set_var, p_for_block)

This function generates the code for the OAL for each statement and places it in the output fragment attribute body . All the fragments associated with the for block have already been processed when this function is called and are contained in the p_for_blck fragment.

The generalization of the OAL is:

for each <p_inst_ref_var> in <p_inst_ref_set_var>

For example: for each dog in dogs

stmt_while (p_condition_rval, p_while_blck)

This function generates the code for the OAL while statement and places it in the output fragment attribute body . The conditional expression for the while statement is in the fragment p_conditional_rval (produced by an rval_* function). All the statements contained in the block associated with the while have been processed when this function is called. The generated code is contained in p_while_blck.body (produced by blck_end function).

The generalization of the OAL is:

while ( <p_condition_rval> )

For example: while ( ack_count <= 5 )

stmt_break

This function generates the code for the OAL break statement and places it in the output fragment attribute body . The generalization of the OAL is:

break;

stmt_continue

This function generates the code for the OAL continue statement and places it in the output fragment attribute body . The generalization of the OAL is:

continue;

stmt_return_void

This function generates the code for the OAL return statement and places it in the output fragment attribute body . The generalization of the OAL is:

return;

stmt_return_rval (p_return_rval)

This function generates the code for the OAL return statement and places it in the output fragment attribute body . The generalization of the OAL is:

return <p_return_rval>;

For example: return param.number*2;

stmt_control (p_cmd)

This function is invoked when a control statement is used in the OAL. The generalization of the OAL is:

control <p_cmd>;

For example: control stop;

stmt_bridge_void (p_ee_kl, p_method_name, p_param)

This function is invoked when the bridge invocation p_method_name for the external entity p_ee_kl is encountered in the OAL. The OAL bridge does not return a value. Note that the parameter list for the bridge, contained in the p_param fragment, has been completely processed prior to the invocation of stmt_bridge_void. The generalization of the OAL is:

[bridge] <p_ee_kl>::<p_method_name>(<p_param>);

For example: EE:method_one(param1:10, param2:20);

stmt_transform_void (p_obj_kl, p_method_name, p_param)

This function is invoked when the class-based operation invocation p_method_name for the class p_obj_kl is encountered in the OAL. The OAL operation does not return a value. Note that the parameter list for the operation, contained in the p_param fragment, has been completely processed prior to the invocation of stmt_transform_void . The generalization of the OAL is:

[transform] <p_obj_kl>::<p_method_name>(<p_param>);

For example: BRAKE::SetBrakingOption(op:Options::Default);

stmt_transform_instance_void (p_obj_inst_ref_var, p_method_name, p_param)

This function is invoked when the instance-based operation invocation p_method_name for the instance p_obj_inst_ref_var is encountered in the OAL. The OAL operation does not return a value. Note that the parameter list for the operation, contained in the p_param fragment, has been completely processed prior to the invocation of stmt_transform_instance_void . The generalization of the OAL is:

[transform] <p_obj_inst_ref_var>.<p_method_name>(<p_param>);

For example: CAR.resetCoordinate(x:0,y:0);

Special Where Clause Translation

In order to allow easier implementation of optimized searches, additional support for where clauses exists that allows the system architect to mark, through marking, certain where clause expressions as "special". When these special where clause expressions are parsed by the code generator, a different set of fragment translation functions are invoked rather than the functions that get invoked during the processing of a where clause that has not been marked as special.

A where specification ( where_spec ) is a character string that generically describes the where clauses for which the architect has a special implementation. Simply put, where_spec is a string that is the same as the where clause in a select statement except that the parameters passed to the special where are specified with question marks ( ? 's).

Specifically, the ? is shorthand for an expression. Anything which is an expression in the OAL can match the ? in the same place in the where clause.

Examples of valid where_specs:

selected.x == ?
selected.x == bridge EE::foo(a: ?)
(selected.x == ?) AND (selected.y == ?)
selected.x
 

Examples of invalid where specs:

selected.? == ... <- 
attribute name not an expression
selected.x ? 7 <- operator not an expression
selected.x == self.i <- self not allowed
 

Examples of valid usage:

where_spec: selected.x == ?
where clause: selected.x == 7 
where clause: selected.x == bridge EE::foo(a:selected.x)
where clause: selected.x == selected.y
 

The following constructs are allowed in the where_spec :

Instance reference variables:

  1. selected

Most model names:

  1. class keyletters
  2. class attributes
  3. EE keyletters
  4. bridge and operation names and data parameter names

Expressions:

  1. constants (integer, real, string, boolean)
  2. complex expressions
Database changes

To provide the architecture developer with the ability to specify a where clause that has a special implementation, the architect must first add a class, called a where specification class, to the metamodel that defines the tables in the xtUML database. The class must have the following attributes:

Attribute Name

Type

obj_kl

string

where_spec

string

key

string

ret_val

boolean

obj_kl are the keyletters of the class in the xtUML model for which the special where clause is being defined.

where_spec is as described above.

key is a string that identifies this special where clause to the architecture. The parser passes this string back to the architecture when a special where clause is found. The key must uniquely identify each special where clause.

ret_val is used by the translation rules to indicate success or failure back to the rule.

The key attribute is an identifier for the where specification class. The attributes obj_kl and where_spec also form an identifier for the class.

The translator doesn't care what the class' name is, only that it has these attributes. For the sake of the following discussion, we'll assume the class is named Special_Where with key letters A_SW .

When the architect wishes to inform the translator that a where clause has a special implementation, he creates an instance of the Special_Where class, and sets the obj_kl , where_spec , and key attributes to the appropriate values.

The class is created by specifying the following table in the metamodel schema file: 

CREATE TABLE A_SW (
 OBJ_KL STRING, 
WHERE_SPEC STRING, 
KEY STRING, 
RET_VAL BOOLEAN);
 

INSERT statements are used to specify the instances of where specification classes to be created in the xtUML database. For example:

 INSERT INTO A_SW
 VALUES (
 'D_DOG',
 'selected.breed == ?',
 'dogbreed',
 FALSE);
 INSERT INTO A_SW
 VALUES (
 'D_DOG',
 'selected.color == ?',
 'dogcolor',
 FALSE);
 

The pt_gen_import command is used to place these instances in the xtUML database.

Rule Commands

The following rule command is used to identify a where specification to the translator:

.special_where add <obj_inst_ref>

Where <obj_inst_ref> is a reference to an instance of the Special_Where with the appropriate data values set. On completion of the command, the attribute ret_val contains the result of the operation ( TRUE for success, FALSE for failure). If <obj_inst_ref> does not have an attribute named ret_val (of type boolean ), the command fails and an error message stating there is no ret_val attribute is output, and the command returns.

Possible errors:

  1. <obj_inst_ref> does not have an attribute obj_kl
  2. <obj_inst_ref> does not have an attribute where_spec
  3. <obj_inst_ref> does not have an attribute key
  4. a special clause already defined for obj_kl or where_spec
  5. a special clause already defined for key
  6. there is an error parsing the where_spec

If the architect no longer wants to treat a where clause as special, the following rule command is used:

.special_where clear <obj_inst_ref>

Where <obj_inst_ref> is a reference to an instance of the Special_Where with the appropriate data values set. The command uses both the key attribute and the obj_kl/ where_spec attributes to search for the special where instance to delete. On completion of the command, the attribute ret_val contains the result of the operation ( TRUE for success, FALSE for failure). If <obj_inst_ref> does not have an attribute named ret_val (of type boolean ), the command fails. An error message stating there is no ret_val attribute is output and the command returns. If no matching special where instance is found, the command sets ret_val to FALSE .

Possible errors:

<obj_inst_ref> does not have the attributes key , obj_kl or where_spec

If the architect no longer wants to consider any where clause as special, the following rule command is used:

.special_where clear all

If there are no special where clauses defined, this command returns.

The architect can determine if a where clause is special using the following statement:

.special_where query <obj_inst_ref>

The function determines if this is a special where clause. The command uses both the key attribute and the obj_kl/where_spec attributes to search for the special where instance to query. If it is, then the ret_val attribute of the <obj_inst_ref> is set to TRUE , otherwise it's set to FALSE . If <obj_inst_ref> has no ret_val attribute, the command fails, and an error message stating there is no ret_val attribute is output, and the command returns.

When an .al_xlate command is reached in the rule file, the parser invokes fragment functions according to whatever special where clauses are (or are not) defined at the time the .al_xlate command is executed.

Examples:

.// A_SW is the key letters for the Special Where class
.select any sw1 from instances of A_SW where selected.key == "dogcolor"
.special_where add sw1
.select any sw2 from instances of A_SW where selected.key == "dogbreed"
.//
.special_where add sw2
.special_where clear sw1
.special_where query sw1
.//
.if (sw1.ret_val == TRUE )
 .// error - shouldn't exist
.end if
.special_where clear all
Fragment Translation Functions

The fragment translation functions invoked for a where clause are different depending on whether the where clause is special or not. In either case, the first fragment translation function invoked is where_begin , and the last fragment function invoked is where_end .

If the where clause is not special, then the where expression is parsed and the appropriate rval_ fragment functions are invoked as would be expected for any other expression. The fragment function where_add_normal is then called with the rval_ fragment that represents the expression contained in the where clause.

If the where clause is special, then the where clause is parsed. First a call to param_begin_where_special is made, then for each question mark in the <where_spec> , a call is made to param_add with the rval that corresponds to the question mark place holder. The p_param_name parameter to param_add is set to the position of the parameter in the original <where_spec> ("1", "2", "3", etc.). After all the parameters are processed, the param_end function is called followed by where_add_special .
