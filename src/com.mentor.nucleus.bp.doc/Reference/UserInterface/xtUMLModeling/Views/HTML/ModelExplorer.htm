Model Explorer View

The Model Explorer view displays a hierarchical view of a UML model.  It also serves as a navigation tool to the various editors associated with model elements.  All editors are accessed via the Model Explorer view.

Packages

Packages provide the primary means of organizing the data in the system.  BridgePoint supports several types of stereotyped packages, each with its own contents.

Each package may contain model elements that belong to the package, or other packages of the same type.  For example, a data type package can contain data types and other data type packages, a use case package can contain use cases and the relationships between them, and other use case packages. 

The following table describes each of the organizational packages.

Package

Description

System

Topmost package.  Contains zero or more domain packages.

Domain

Contains zero or more of the following packages: data type, function, external entity, sequence, use case, communication, activity, and subsystem.  A domain package may only contain other packages, not individual model elements.

Data Type

Organizes data types for the xtUML model.  A data type package contains individual data types and enumerations, or other data type packages.  When a new xtUML model is created, a package called Datatypes is created which contains the primitive, or core data types such as: boolean, date, integer, and so on.

Function

Organizes functions in the domain.  A function package contains function packages.  Functions themselves do not have a graphical notation.

External Entity

Organizes external entities in the domain.  An external entity package contains external entities (and their bridge operations and events), or other external entity packages.

Subsystem

Organizes classes in a subsystem.  Subsystem packages contain classes, or other subsystem packages. 

Event Data Item

Organizes the event data items for a state machine.  There is one event data item package per state machine.

Sequence

Describes interactions over time between model elements.   Nominally the roles of objects are drawn in columns arranged horizontally with the vertical axis representing time.  Sequence packages may be placed in Domain and Subsystem packages or nested inside other Sequence packages.

Use Case

Describes relationships between use cases and actors.  Use case packages may be placed in Domain and Subsystem packages or nested inside other Use Case packages.

Communication

Describes the communication paths between model elements.  Communication packages may be placed in Domain and Subsystem packages or nested inside other Communication packages.

Activity

Describes the procedural flows between activities.  Activity packages may be placed in Domain and Subsystem packages or nested inside other Activity packages.

BridgePoint packages allow you to organize the model elements that comprise the system.   BridgePoint enforces model element groupings similar to those of UML 2.0.  Examples of groupings and the model elements contained within them are as follows:

1.       Data type package - contains primitive, user-defined, and enumeration data types.

2.       Subsystem package - contains classes, associations, and other subsystem packages.

Model Explorer Tools

The Model Explorer view contains the following tool buttons which help you navigate and manage your model data. 

 Collapse All

Collapses all the expanded directories in the Model Explorer to the highest model element in the hierarchy.

 Link with Editor

Synchronizes an open diagram with the currently selected model element in the Model Explorer. 

Filters

You can also control which model elements are displayed in the Model Explorer view by choosing to hide the elements you do not want to view.  To do this, in the Model Explorer view, click the down arrow  and select Filters from the drop-down menu. The Model Explorer Filters dialog box appears.

Select the model elements you wish to exclude from Model Explorer view and click OK.

Creating and Deleting Model Elements

Certain model elements can be created and deleted from the Model Explorer view.  Some model elements, such as relationships between classes or transitions between states, can only be created on a Diagram Editor.  

To create a new model element, right-click the enclosing model element and click New > <Model Element Name>.  

A new model element is created in the tree with the name Unnamed <Model Element Name>.

To delete a model element right-click the model element and select Delete.

Renaming Model Elements

Model elements have in-place renaming in the Model Explorer view.  To rename an element, right-click the element and select Rename. 

Editors for Model Elements

To open the default editor for a model element, double-click on that element in the Model Explorer view. To open the model element in a different editor, right-click the model element, and click Open With. Then, select the desired editor from the list.

In the Model Explorer view for the MicrowaveOven sample model displayed below, for example, you can double-click the Microwave Oven subsystem package to view a Class Diagram Editor.  You can double-click the Oven class to open its Description Editor and you can double-click a function to open its Activity Editor.

Activity Editor

The default editor for the following model elements is the Activity Editor.

·         Function

·         Bridge operation

·         Derived attribute

·         Operation (class-based or instance-based)

·         State action

Description Editor

The default editor for all other model elements, e.g., classes, attributes, and parameters is the Description Editor. If you double-click a model element that does not have an associated Diagram Editor, an Activity Editor appears if the model element can have activities described by action language. Otherwise, a Description Editor appears.
