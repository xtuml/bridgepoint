State Machine Package

A State Machine Package is used to document states, events, transitions, and activities that define the behavior for a class.  A class can have to state machine packages, an Instance State Machine Package which defines the behavior for each instance of the class, and a Class State Machine Package, which defines the behavior for the class.

State

A state is a stable stage in the lifecycle of the class. A class may specify progression through a number of states. Each time, progression is triggered by an event and specified by the transition to which the event is assigned.

Graphical Notation

A state is represented by a rectangle with rounded corners.  Each state has to compartments.  The first contains the number and name of the state.  The second contains the activities executed upon entry into the state.

Event

An event is an abstraction of significant occurrence in the lifecycle of a class. Events cause the transitions from one state to another in a state machine diagram. In fact, the only place you can define an event is on the state machine diagram. Also, you can only define an event to one state machine. Events created in the state machine of a supertype can be assigned to transitions on the state machine of subtypes.

Every event defined on the state machine of a supertype is considered to be inheritable by default, so you do not have to declare an event to be inheritable.

Graphical Notation

An event is shown as a text label adorning on a transition.

Using Events

Events which are defined and not already in use on a supertype’s state machine may be assigned to a transition of a subtype’s state machine based on the following two rules:

§         If an event is assigned to a transition in a state machine of some supertype, it is not available for assignment on transitions of the state machines of any of its subtypes.

§         If an event defined at the supertype level is assigned to a transition on the state machine of at least one subtype, the event is not available for assignment on a transition of any supertype’s state machine.

Renaming Events

You may only rename an event within the state chart in which it is defined. You can not rename events in a subtype state machine.

Transition

A transition represents a progression of an instance from one state to another.

Graphical Notation

A transition is a uni-directional path between two states denoting a flow of control from the source state to the destination state. A transition is always labeled with the event that causes the transition from the source to the destination state.

Transitions may show a path between two distinct states, or the same state, the latter referred to as a self-transition. Several transitions may cause entry into the same state, the only restriction being that each transition must be labeled with a unique event and each transition into a state must carry the same event parameters.

Activity

An activity is an executable specification of processing that acts on one or more model elements in the domain. What an instance does when it transitions into a state is specified by one or more actions. Actions can do many things: create and delete instances, select instances from the class extent, select instances across relationships, relate and unrelate instances, assign values, compare values, and invoke operations in other domains.

Graphical Notation

Activities are shown inside a state symbol.
